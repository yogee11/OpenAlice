# Workspace Template Upgrade

This guide owns the in-place reconciliation of launcher-managed template
assets into an existing durable Workspace. It defines the three-way merge,
review contract, transaction and recovery rules, and the boundary shared with
a future Workspace Merge/Absorb workflow.

Related guides: [[docs/workspace-agent-guidance.md]],
[[docs/workspace-lifecycle.md]], and [[docs/project-structure.md]].

## Product Promise

Template Upgrade means “bring the template's newer core assets to this desk
without erasing the work already done here.” It is not a second bootstrap and
never replaces the complete Workspace directory.

Only a template that explicitly declares
`"upgradeStrategy": "managed-context"` participates. Chat opts in. Auto-Quant
and third-party templates remain outside this generic path until they declare a
safe, template-specific asset boundary.

The current managed set is intentionally narrow:

- `README.md`, `AGENTS.md`, and `CLAUDE.md`;
- `.agents/skills/**` and `.claude/skills/**`;
- legacy `.pi/skills/**`, so an unchanged duplicate skill tree can be removed.

Research, reports, Issues, Inbox records, credentials, Git history, runtime
state, and every other Workspace file are never candidates for Template
Upgrade.

## Three-Way Change Plan

Every plan compares three exact snapshots:

```text
Base       last template snapshot applied to this Workspace
Local      managed files in the live Workspace now
Incoming   managed files generated by the current template for this Workspace
```

Files are classified by fingerprints, not timestamps:

| Classification | Meaning | Default action |
|---|---|---|
| Ready | Incoming changed; Local still matches Base | Apply Incoming |
| Preserved | Local changed; Incoming still matches Base | Keep Local |
| Conflict | Local and Incoming both changed | Require an explicit per-file choice |
| Unchanged | Local already equals Incoming | Do nothing |

The preview digest covers Base, Local, Incoming, template identity, and both
versions. Apply must present that digest. The manager materializes Incoming
once for an apply attempt; the snapshot validated by the digest is the exact
snapshot written to disk. A Local or template change after preview therefore
returns a stale-plan error instead of silently changing the reviewed operation.

## Baseline and Version State

New Workspaces record their exact managed snapshot immediately after the root
Git commit:

```text
<workspace>/.alice/template-upgrade/
├── state.json          applied template and version
└── baseline.json.gz    exact managed Base snapshot
```

This directory is added to `.git/info/exclude`: it is launcher bookkeeping,
not part of the user's work log. It contains no credentials.

Legacy Workspaces do not need a bulk migration. On first preview OpenAlice
reconstructs Base from the Workspace root commit and uses
`spawnedFromVersion` as its lineage version. After the first successful
upgrade, the recorded baseline becomes authoritative. Editing README
frontmatter is not equivalent to completing an upgrade and cannot dismiss the
upgrade signal.

## Apply Transaction

Apply takes the shared checkout-operation lease and is serialized per Workspace.
Offboarding uses the same lease; a future Merge/Absorb operation must do so as
well, so directory reconciliation and directory moves cannot race. Apply also
refuses to start while an interactive
Session, WebPi Session, or headless run is active. It also refuses an already
staged Git index so the template change cannot absorb an unrelated staged
change.

The transaction is:

1. re-plan against one Incoming snapshot and validate the reviewed digest;
2. validate every conflict resolution;
3. write a journal plus compressed Before and Incoming snapshots under
   `.alice/template-upgrade/transaction/`;
4. atomically replace or remove only selected managed paths;
5. stage only those paths and create an isolated
   `template(<name>): upgrade <old> -> <new>` commit;
6. persist Incoming as the next Base and remove the transaction directory.

On failure before commit, recovery restores every touched path and resets only
those paths in the index. If the commit completed but the process stopped
before state persistence, startup recognizes the commit trailer, finalizes the
new baseline, and removes the journal. Recovery runs before scheduled work is
started, so agents never begin against a half-applied instruction tree.

## User Surface

An upgrade badge opens Workspace Settings -> Template. The review speaks in
user terms—ready updates, protected customizations, and files needing a choice.
Apply remains disabled until every conflict has a decision. The operation ends
with the Git commit id, making the change inspectable and reversible through
normal Workspace history.

The same transaction is available from inside the current Workspace without
hand-authoring API calls:

```bash
alice-workspace template upgrade          # read-only preview
alice-workspace template upgrade --apply  # re-plan and apply the exact current plan
alice-workspace template upgrade --id <workspaceId>          # preview a peer
alice-workspace template upgrade --id <workspaceId> --apply  # upgrade a paused peer
```

Conflicts require one repeatable `--keep-workspace <path>` or
`--use-template <path>` decision per path. The caller's Workspace is the
default target; an interactive manager can pass `--id` to reconcile a paused
peer without entering its Session. The launcher still owns the plan digest
internally. This keeps the shell surface aligned with the UI's safety contract
while avoiding HTTP and transaction plumbing in agent prompts. A live Session
cannot upgrade its own instructions underneath itself; pause it or apply from a
separate manager Workspace.

Cross-Workspace apply is an interactive management action. A headless run may
preview another desk but the tool rejects applying that peer's upgrade.

## Shared Foundation, Different Workflows

Template Upgrade and a future Workspace Merge/Absorb are not the same product
operation:

- Upgrade has an unoccupied template as Incoming and a recorded template
  snapshot as Base. It changes only launcher-managed assets.
- Merge has another worked-in Workspace as a source. It must define ownership,
  Session/resumeId retirement or transfer, Issue and artifact provenance, Git
  history policy, and source-Workspace offboarding.

They should share the source-neutral mechanics—asset inventory, fingerprinted
snapshots, change planning, conflict choices, reviewed digest, serialized
transaction, recovery, and audit commit—but not share a single business API or
pretend that merging coworkers is merely a template update.

## Load-Bearing Code

- `src/workspaces/template-upgrade.ts` — snapshot, planning, transaction, and
  recovery.
- `src/workspaces/workspace-operation-guard.ts` — process-local serialization
  shared with offboarding and future checkout-wide operations.
- `src/workspaces/template-registry.ts` — explicit template opt-in.
- `src/workspaces/workspace-creator.ts` — exact creation baseline.
- `src/workspaces/service.ts` — composition, busy-state gate, and recovery
  ordering.
- `src/webui/routes/workspaces.ts` — preview and apply API.
- `ui/src/components/workspace/WorkspaceTemplateUpgradePanel.tsx` — review and
  conflict decisions.
- `src/workspaces/template-upgrade.spec.ts` — classification, stale preview,
  concurrency, rollback, baseline, and real-template materialization coverage.
